Quickshell on NixOS

00Summary & Verdict

Quickshell is a QtQuick toolkit for building the entire desktop shell — bars, launchers, notifications, lock screens, OSD, even a display manager — in one declarative QML configuration that hot-reloads on save. evo-x2 currently stitches together nine separate tools (Waybar, Rofi, Dunst, Swaylock, Wlogout, Swayidle, wl-clipboard/cliphist, SDDM, polkit_gnome) across five different config languages (CSS+JSON, Rasi, libnotify KDL, KDL, shell scripts). Quickshell could collapse most of that into a single, themable, reactive QML codebase wired directly to SystemNix's own services.

The opportunity is large, the tooling is real, and the Nix story is clean. The honest catch: Niri has no first-class Quickshell module, Quickshell is pre-1.0, and QML is a real programming language — not a config format.

The bottom line Worth pursuing — but via a hybrid path. Adopt an existing Niri-native Quickshell shell (iNiR or DankMaterialShell) to validate the stack, then carve out a custom shell that draws live data from this system's services (Gatus, SigNoz, DNS-blocker, dual-WAN, GPU/NPU, BTRFS). That's what makes it exceptional rather than another dotfiles clone.
One action that unlocks the most value Add the quickshell flake input and stand up a minimal bar that reads Niri workspaces via the qml-niri plugin. Prove the hot-reload loop before building anything ambitious.

01What Quickshell Actually Is

Quickshell is not a status bar. It's a toolkit for building status bars — and launchers, lockers, notifications, and greeters. It's based on QtQuick and configured in QML. Crucially, configuring it is closer to writing an application than editing a config file. That is both its power and its cost.

The superpowers

+
Hot reload on save Save a QML file and the shell updates instantly — no restart, no logout. Iterate as fast as you can type.
+
One language, one brain The entire shell — bar, notifications, locker, launcher — lives in one reactive QML tree sharing state, theme, and IPC.
+
An LSP for QML qmlls gives autocompletion, go-to-definition, and hover docs — including Quickshell's own types.
+
Wayland-native, not a hack First-class wlr-layer-shell, session lock, foreign-toplevel, and idle protocol support — not screensaver-window tricks.

The integration surface

These ship in the box — no shell scripts, no polling daemons, no third-party glue required to reach them from QML:

The single most important row is Quickshell.Io. It's the escape hatch that lets a QML shell talk to anything — including Niri's IPC and every SystemNix service. That is what makes a custom shell viable here.

04Everything Cool We Could Build

This is the creative core. The differentiator isn't "another pretty bar" — it's a shell that reads live state from this system's own services and composes them into one reactive surface. Grouped into five pillars.

Pillar A — A unified reactive bar

One PanelWindow that replaces Waybar and its five hand-rolled scripts. Because Quickshell binds reactively, modules update the instant state changes — no polling, no JSON parsing, no exec scripts.

Niri workspaces & columns

Live workspace strip + focused window title, fed by the qml-niri plugin or niri msg event stream.

Media with art

Now-playing title, cover art, and controls from the MPRIS service — replaces waybar-media.

Audio with live meters

Per-sink/source VU meters from Pipewire peak monitors — something Waybar simply cannot do.

Network & connectivity

Wi-Fi SSID, signal, wired link state, and connectivity from NetworkManager — plus a scan picker.

Power & profiles

Battery + a one-click power-profile switcher (performance / balanced / power-saver).

System tray done right

A real StatusNotifierItem host — the feature Waybar struggles with most.

Pillar B — Mission Control (the OSD / quick-settings)

A PopupWindow — think GNOME's quick settings or a phone's control center — that aggregates every toggle and slider in one animated panel: volume + brightness sliders with live meters, Wi-Fi/Bluetooth pickers, power-profile radios, a Do-Not-Disturb switch, a Caffeine (idle-inhibit) toggle, night-light, and the now-playing widget with scrubbing. Pillar C — Notifications, lock, and power — unified

Replace three disjoint tools (Dunst, Swaylock, Wlogout) with one themed family:

Rich notification daemon

Banners with images, app icons, action buttons, and inline reply. A swipe-down center shows grouped history — no more piping dunstctl through Rofi.

Session lock with a soul

WlSessionLock + PAM: a clock, the now-playing track, blur backdrop, and proper authentication — themed identically to the bar.

Power overlay

In-shell polkit

PolkitAgent prompts that match the desktop instead of a GTK island.

Pillar E — Signature features (the things no upstream shell has)

These are the widgets that make this shell uniquely SystemNix. Every service below already runs on evo-x2 with an HTTP or IPC surface — Quickshell's Quickshell.Io reads them directly into reactive widgets. No other dotfiles shell can do this, because no other dotfiles shell knows about your services.

AI Brain Monitor — Ollama :11434

Poll /api/ps to show the currently loaded model, its VRAM footprint, and tokens/sec. Know at a glance whether llama3.1:8b is hot or if the GPU is free. Click to swap models from a popup. 

// Real QML — reads Ollama's live state
Process {
  command: ["curl", "-s",
    "localhost:${ports.ollama}/api/ps"]
  stdout: SplitParser {
    onRead: line => {
      const d = JSON.parse(line)
      aiText.text = d.models?.[0]?.name ?? "idle"
      vram.value  = d.models?.[0]?.size_vram ?? 0
    }
  }
}

This shell would be the only one in the world that shows your AI model's VRAM, your ad-blocker stats, your task radar, and your Minecraft server — because it's wired to this machine.

Niri exposes column/tile geometry — render live mini-previews of windows in an overview, a feature unique to scrollable tiling.

IdleInhibitor as a one-tap toggle; a "focus" mode that mutes notifications and pins a workspace.

A day in the life

You unlock the screen. Immich has loaded your "on this day" photo behind the clock — not a stock wallpaper. The bar shows 3 pending high-priority tasks from Taskchampion, pulsing gently. Your Ollama model is warm at 6.2 G VRAM; the bar knows before you do.

Mid-morning, the DNS-blocker count ticks past 2,000 ads blocked today — a tiny green badge. A SigNoz dot turns amber: a service is degraded. You click it, and a popup shows the failing endpoint without opening a browser. You fix it, the dot goes green, the bar settles.

You start a voice agent session. The bar lights up with a live waveform and real-time transcription from Whisper. Minecraft players tick up to 4/20 in a subtle counter. You press a key, the Mission Control OSD slides down — you switch to power-saver, mute notifications (focus mode), and inhibit idle. The whole shell dims slightly to acknowledge the change. One brain. One theme. No tools out of sync.

05The Niri Reality — Be Honest About the Gap

Quickshell has native compositor modules for Hyprland and i3/Sway. It does not have one for Niri, and Niri does not implement the i3 IPC. This is the single most important technical constraint. But it is solved — there are three viable paths.

Recommended

The qml-niri plugin

github.com/imiric/qml-niri · needs Quickshell ≥0.3.0, Niri ≥26.04

A QML plugin wrapping Niri's IPC into native Qt types: WorkspaceModel, WindowModel, and focusedWindow. Its flake even ships a Quickshell build with the plugin pre-loaded. The cleanest path to a native feel.

Most control

Direct socket to $NIRI_SOCKET

github.com/snowarch/iNiR · the approach full Niri shells use

Open a persistent connection to Niri's socket, send "EventStream", and parse newline-delimited JSON for workspace/window state. More code, but full access to Niri-specific data like tile geometry (enables live previews).

Fallback

Quickshell.Io.Process + niri msg

e.g. niri msg -j workspaces

Spawn niri msg and parse JSON. Simple, but one-shot — you'd have to poll or combine with the event stream for live updates. Fine for occasional queries, not for a reactive bar.

Good news Window titles work out of the box via Quickshell.Wayland.ToplevelManager (Niri supports wlr_foreign_toplevel). And there are already complete Niri+Quickshell shells to learn from — iNiR (bar, dock, overview, notifications, lock, settings) and DankMaterialShell (modular, supports Niri + Hyprland).

06How It Slots Into the SystemNix Flake

The Nix story is clean and fits the existing flake-parts conventions almost exactly. No new patterns required — just the standard "add an input, add a service module" flow this repo already uses.

1

Add the flake input (with follows)

Track the upstream flake, and must follow nixpkgs so Qt versions stay in lockstep — mismatched Qt causes crashes. 

quickshell = {
  url = "git+https://git.outfoxxed.me/outfoxxed/quickshell";
  # MUST follow nixpkgs — mismatched Qt deps crash at runtime
  inputs.nixpkgs.follows = "nixpkgs";
};
2

Build the package, optional extra modules

Use .withModules […] if the config needs the qml-niri plugin or Qt5Compat (for gaussian blur); plain .default otherwise. 

qsPkg = quickshell.packages.${system}.default;# .withModules [ qml-niri ]
3

Ship the QML config declaratively

Either vendor the QML under modules/nixos/services/quickshell/ and symlink, or write it into the store and launch via a systemd user service (the established pattern for the existing HM tools).

4

Tie into existing single sources of truth

Theme from theme.font.mono / colorScheme.slug (Catppuccin Mocha already declared). Derive any ports from ports.*. This keeps AGENTS.md's "never hardcode" rules intact.

5

Launch from Niri, not from the shell

Start via spawn-at-startup "quickshell" "--config" … in niri config — keeping the compositor as the session owner, consistent with how Waybar is started today.

Where it lives Most cleanly as a Home-Manager module under platforms/nixos/desktop/quickshell.nix (next to waybar.nix), enabled in home.nix. A flake-parts service module would only be needed if it ever exposes system-level options or ports.
Hot-reload dev loop — the killer workflow During development, run quickshell against a config path outside the Nix store (e.g. a git-tracked checkout). Edit QML → save → the shell updates instantly. No just switch, no rebuild, no logout. This is the fastest UI iteration loop this desktop has ever had.

07Build vs Buy — Don't Reinvent the Wheel First

The Quickshell community already ships complete, beautiful shells. Before writing thousands of lines of QML, weigh the options. The recommendation is not "build from zero" — it's "stand on a proven shell, then diverge where it matters."

08Risks, Costs & Tradeoffs

Real · Must accept

Pre-1.0 with promised breaking changes

Quickshell v0.3.0 · migration guides promised before 1.0

The toolkit explicitly states there will be breaking changes before 1.0. Pin a tag (?ref=) rather than tracking master, and budget for occasional migrations. Mitigate by keeping the old stack working in parallel during rollout.

Real · Skill cost

QML is programming, not config

affects maintenance burden

A custom shell is an application with state, bindings, and async IO. It's far more powerful than Waybar's JSON, but also a real codebase to maintain. This is the core tradeoff: capability vs surface area. Mitigate by starting from a community shell.

Managed · External dependency

Niri integration depends on a third-party plugin

qml-niri · not part of Quickshell core

The cleanest Niri path rides on qml-niri, an independent project. If it stalls, you fall back to the socket approach (which iNiR already proves works). Not a blocker — but a dependency to track.

Non-issue · NixOS-only

Not relevant to Darwin

macOS is unaffected

Quickshell is a Wayland/X11 shell tool. It belongs only on evo-x2. Darwin (Lars-MacBook-Air) keeps its minimal HM setup untouched — no disk/perf impact on the constrained laptop.

09Phased Roadmap

A staged path that de-risks at every step and never leaves the desktop broken. Effort estimates assume one developer comfortable with Nix but new to QML.

P0

Spike — prove the loop · ~2h

Add the flake input, run Quickshell with a 30-line bar that prints the clock and a Niri workspace read via niri msg. Confirm hot-reload. Decide go/no-go based on the dev experience.

P1

Adopt a base shell · ~1 weekend (8–12h)

Fork iNiR or DankMaterialShell into the repo, theme to Catppuccin Mocha, wire it as an HM module. Run it alongside the existing Waybar/Dunst to compare. Keep Swaylock/Swayidle for now.

P2

Cut over the easy wins · ~1 week (5–10h)

Replace Dunst → NotificationServer, Wlogout → power overlay, polkit_gnome → PolkitAgent, add the SystemTray. Remove the now-dead shell scripts from waybar.nix.

P3

The SystemNix-native layer · the differentiator · ~10–20h

Build the Pillar D + Pillar E widgets: DNS stats, dual-WAN/failover, Gatus/SigNoz health dots, BTRFS freshness, GPU/NPU, Ollama model status, Taskchampion radar, voice agent waveform, Immich memories. This is where the shell becomes unique to this machine.

P4

Lock screen & polish · ~5–10h

Build the WlSessionLock+PAM lock screen and Mission Control OSD. Add dynamic wallpaper theming via ColorQuantizer. Retire Waybar entirely.

P5

Optional: a greetd greeter · stretch · ~20h+

Only once the in-session shell is proven — a custom display manager on greetd. High effort, security-sensitive; treat as a stretch goal.
