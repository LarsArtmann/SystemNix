00Summary

SystemNix is a mature, heavily-documented NixOS configuration managing a single x86_64 server (evo-x2, 128 GB RAM, AMD Ryzen AI Max+ 395, 2 TB NVMe) and a minimal macOS laptop (Lars-MacBook-Air, aarch64-darwin, 24 GB / 256 GB). The codebase is clean — zero inline TODO/FIXME markers; all deferred work lives in TODO_LIST.md and ROADMAP.md.

The system runs 27 services, a 13-plugin DankMaterialShell desktop shell, a custom DNS blocker (~930 lines Go), and full observability (SigNoz + Prometheus + Gatus with Discord alerting). Forgejo OIDC SSO via Pocket ID was just implemented this session (uncommitted — 4 files, 109 insertions).

The three biggest risks right now
  1. Root disk at ~93% — high risk of repeating the 2026-06-26 BTRFS metadata ENOSPC crash
  2. /data subvolume has zero snapshots and zero backup — Immich media, Docker volumes, AI models all unprotected
  3. No offsite backup exists anywhere — a disk failure or theft loses everything except DiscordSync's Turso cloud sync

DTotally Fucked Up

Critical Root disk at ~93% — ENOSPC imminent

evo-x2 root BTRFS

This is the exact precondition that caused the 2026-06-26 BTRFS metadata ENOSPC crash (I/O deadlock → WDT hard reset). df reports Data-pool free space, NOT chunk-level allocation — the monitoring stack was blind then and the btrfs-health.nix guard is the mitigation. But at 93%, the guard will start aborting nix-gc, meaning garbage can't be collected, meaning disk fills faster. This is a negative feedback loop.

Immediate action needed Manual cleanup: sudo rm -rf /nix/var/nix/builds/nix-* (stale OOM sandboxes), expire old BTRFS snapshots, run nix-collect-garbage --delete-older-than 7d. Consider growing the partition.

Critical /data — no snapshots, no backup, biggest data store

/data (BTRFS toplevel, subvolid=5)

Immich media library (irreplaceable photos/videos), Docker volumes (service state), AI model cache (100+ GB), containerd metadata. None of it is snapshotted or backed up. A disk failure = total, irreversible loss of all photos and service state. This is the single most dangerous gap in the entire system.

Critical Zero offsite backup anywhere

Entire system

Every backup that exists (btrbk root snapshots, Forgejo dump, Immich/Twenty/Manifest pg_dump) is local to the same disk. Disk failure, theft, fire, or physical damage = total loss. DiscordSync's Turso cloud sync is the only offsite replication, and it's just message history.

G — The One Question I Cannot Answer

Why are deploys being deferred when fixes are committed?

Multiple status reports (2026-06-28, 2026-06-29) describe a recurring pattern: "73 commits undeployed", "zero deploys", fixes committed to master but not applied to the running evo-x2 system. The pre-deploy-check pipeline is excellent and the deploy script (nix run .#deploy) is well-engineered. Yet deploys accumulate a large backlog before being applied.

I cannot determine from the code alone what is creating the friction. Is it build time? (ROCm/CUDA derivations can take 30+ min.) Is it fear of the switch-to-configuration start-limit issue? Is it the disk pressure making builds risky? Is it a workflow preference (batch deploys)? Or is evo-x2 sometimes unreachable (network/sleep)?

Understanding the root cause would unlock all the undeployed fixes — including this session's Forgejo OIDC SSO work. If I knew the constraint, I could either remove it (e.g., split heavy derivations, add a remote builder) or design a lighter-weight deploy path for config-only changes.
