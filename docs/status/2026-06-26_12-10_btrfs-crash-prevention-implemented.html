06Totally Fucked Up

🔴 The 2026-06-26 Crash Itself (Recovered)

Boot -1: 1cc421c712724ed5b23fa899ba55d577 — Jun 23 05:50 → Jun 26 00:02

The nightly nix-gc timer fired at 00:00 on a filesystem with 1 MiB device-unallocated. GC's metadata transactions deadlocked the filesystem. The kernel froze in D-state I/O wait. The sp5100-tco hardware watchdog fired at ~00:02:30 — destructive raw reset. No pstore dump (WDT bypassed kernel panic handlers). System couldn't boot normally (activation is also metadata-intensive). Required USB installer + partition grow to recover.

This was the 2nd time. First: 2026-06-15 (manual trigger). This time: fully automated, structural. Would have recurred every night under the same conditions. Now prevented by the GC guard.

🔴 The Original Parser Bug (Fixed Before Push)

Initial implementation had an AWK function that used {...} block instead of BEGIN {...} — meaning it would have returned empty strings for all byte values. The GC guard would have seen 0 for device-unallocated (triggering fail-open default of 100%), and the metrics would have reported zeros. Caught during self-review, fixed by switching to --raw flag entirely.

07What to Improve

1. Deploy the prevention stack

The code is committed but not deployed. The system is currently running without the GC guard. This is the single highest-priority action.

2. Run a metadata balance

Metadata at 92% is above the 85% warning threshold. A metadata-only balance would consolidate chunks and bring utilization down. Safe now that device-unallocated is at 27%.

3. Add BTRFS check to pre-deploy-check

The nix run .#pre-deploy-check script currently checks for boot-breaking mount options but not BTRFS chunk allocation state. Adding a device-unallocated check would prevent deploying to a system on the edge.

4. Consider nowayout=1 for the hardware watchdog

With nowayout=0, the WDT is disarmed when no process holds /dev/watchdog0 open. This means boot/activation hangs have NO watchdog protection — the system hangs forever. nowayout=1 would keep the WDT armed during boot, but risks resetting during slow BTRFS recovery. The tradeoff needs careful evaluation.

5. Snapshot retention could be tighter

Current retention: snapshot_preserve_min = 7d; snapshot_preserve = 14d 4w. The crash analysis noted 8 daily snapshots pinning extents via CoW. Consider reducing to 3-4 daily snapshots to reduce metadata pressure.
