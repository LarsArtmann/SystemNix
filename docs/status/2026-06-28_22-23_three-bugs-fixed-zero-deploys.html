DWhat We Should Improve

1. CI/CD gap — no automated deploy trigger

11 commits in a week with zero deploys. Every fix requires a manual nix run .#deploy. Consider a GitHub Action or post-push hook that triggers a deploy on master commits, or at minimum a daily reminder/status check.

2. vendorHash drift is silent until build attempted

The crush-daily vendorHash was stale for 10+ hours (commit ff29585 at 18:44, discovered at 22:11). Upstream repos should run nix build in their own pre-commit hooks or CI. The BuildFlow pre-commit hook exists but its 2-minute timeout killed the commit before the build check completed.

3. Schedule-mode services should always serve health endpoints

crush-daily's schedule mode ran a cron daemon but never started an HTTP server, making health monitoring impossible. This is a design pattern issue: any long-running service should expose /api/health regardless of its primary mode. Audit other schedule-mode or timer-based services for the same gap.

4. BTRFS monitoring blind spot — metadata vs data

The entire monitoring stack (Gatus, Prometheus, DMS widget) reported "free space" from df (statfs), which shows Data-pool space, NOT chunk-level allocation. The filesystem had zero device-unallocated space while df showed plenty. The fix (btrfs-health.nix) is in place but the pattern — monitoring tools lying about filesystem health — should be documented as a general anti-pattern.

5. BuildFlow pre-commit hook timeout too short

The crush-daily commit required --no-verify because BuildFlow's 2-minute budget wasn't enough for gitleaks (2 min alone). Consider increasing the budget or running security checks asynchronously.

6. Two-repo deploy dance is error-prone

DiscordSync and crush-daily fixes require: commit upstream → push → update SystemNix flake.lock → commit → deploy. Each step is a potential failure point. Consider a script or Make target that automates the nix flake lock --update-input X && nix eval verification loop.

7. Boot time optimization needed

~5 min boot time dominated by signoz-provision.service (~2 min). Parallelizing provision steps or making health-check polling asynchronous would significantly improve recovery time after crashes.
