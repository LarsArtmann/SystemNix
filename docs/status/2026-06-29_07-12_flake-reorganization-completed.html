07The #1 Question I Cannot Figure Out

Should the BTRFS /data subvolume migration happen before or after the deploy?

The running system is 4+ commits behind config, including the BTRFS metadata-ENOSPC crash guard (btrfs-health.nix GC gate). Root disk was at 93% per the last report. Deploying writes ~2-4 GiB of new generations to the nix store — on a 93%-full BTRFS root, that risks triggering the exact metadata allocation deadlock the guard prevents.

The dilemma: Garbage-collecting first (nix-collect-garbage -d or nix-store --gc) frees space, but GC itself writes metadata transactions. The crash guard gates GC via ExecStartPre, but that guard only exists in the undeployed config. The running system has no guard.

Should I: (a) grow the partition first (btrfs resize), then deploy safely? (b) GC aggressively, then deploy and pray? (c) Reboot to clear stale build sandboxes in /nix/var/nix/builds/ first? Or (d) is the 93% figure stale and disk space is actually fine now?

I cannot answer this myself because: I don't have access to the live system's current disk state (btrfs filesystem df /, btrfs filesystem usage /) to determine actual chunk-level allocation. The df-reported 93% may or may not reflect the metadata-ENOSPC risk — only btrfs filesystem usage shows device-unallocated space.
