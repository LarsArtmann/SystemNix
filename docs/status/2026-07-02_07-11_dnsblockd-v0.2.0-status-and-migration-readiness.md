4. **Fix the sqlc/gofumpt cycle** — Either configure sqlc to generate gofumpt-compliant code, or exclude generated files from gofumpt. The perpetual dirty diff is noise that erodes trust in `git status`
5. **Fix `unparam` lint warning** — `dns_observer.go:55` `allowed` param always false. Either wire it or remove it

### SystemNix Integration

6. **Pin dnsblockd flake input to `v0.2.0`** — Currently `ref=master` means any push to dnsblockd master changes the SystemNix build. Pin to tag after migration
7. **Reduce unbound coupling** — 6+ services explicitly depend on `unbound.service`. After migration, these need to depend on `dnsblockd.service` instead (or better: depend on a generic `dns-resolver.service` abstraction)
8. **Mac CA distribution** — `dnsblockd-CA` not in Mac System Keychain. NixOS auto-imports via `security.pki` + NSS `certutil` + Firefox policy. Mac has zero equivalent. Unrelated to migration but blocks HTTPS block page trust on macOS

### Testing & Validation

9. **Integration test for full DNS chain** — Unit tests cover individual components but there's no test that verifies: query → ACL check → local zone → blocklist match → upstream forward → cache → response, end-to-end
10. **SystemNix VM test for DNS migration** — `tests/default.nix` has a basic unbound test. Needs updating for dnsblockd-native DNS
11. **Coverage for `otel` (56.4%) and `cmd/dnsblockd` (59.7%)** — These are the weakest packages. otel needs collector mocks, cmd needs more integration test paths

## g) The #1 Question I Cannot Answer Myself

**How should the SystemNix `dns-blocker.nix` module handle the transition period where `dnsEnable = false` (current: unbound on :53, dnsblockd as HTTP only) coexists with `dnsEnable = true` (dnsblockd on :53, no unbound)?**

This is a Nix module design question with real tradeoffs:

**Option A — Breaking change:** Remove all unbound config from the module entirely. `dnsEnable = true` is the only mode. Clean, but any rollback requires reverting the commit.

**Option B — Feature-flagged dual mode:** Keep both code paths. When `dnsEnable = false`, generate unbound config (current behavior). When `dnsEnable = true`, skip unbound and add DNS fields to YAML. Safe rollback, but doubles the module complexity and maintenance burden.

**Option C — Separate module:** Create `dnsblockd-resolver.nix` as a new module. Keep `dns-blocker.nix` as-is (HTTP-only). Import one or the other per host. Clean separation but fragments config.

**Why I can't decide:** This is a SystemNix architecture call that depends on how Lars wants to handle rollback safety vs. module simplicity. The AGENTS.md says "Breaking changes need explicit user approval." I lean toward **Option A** (clean break) because the unbound config is preserved in git history for rollback, and dual-mode modules become unmaintainable. But this is Lars's call.
