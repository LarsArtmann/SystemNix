### Key Technical Insight Discovered

**`proxyVendor = true` in nixpkgs 26.11:**
- go-modules derivation: runs `go mod download` (has network access via fixed-output)
- Main build: sets `GOPROXY=file://$goModules` (NOT `GOPROXY=off`)
- Main build: does NOT set `-mod=vendor` тАФ no vendor/modules.txt validation
- **Result:** `go mod tidy` CAN run in main build's `preBuild` because GOPROXY points to local cache

**Pattern for workspace Go repos with `mkPreparedSource`:**
```nix
buildGoModule {
  inherit vendorHash proxyVendor;  # MUST inherit proxyVendor!
  overrideModAttrs = _: { preBuild = "go mod tidy"; };  # go-modules derivation (network OK)
  preBuild = "go mod tidy";  # main build (uses GOPROXY=file:// cache)
};
```

**Pattern for simple Go repos with private deps:**
```nix
buildGoModule {
  vendorHash = null;
  deleteVendor = false;  # Keep committed vendor/
};
```

## e) WHAT WE SHOULD IMPROVE ЁЯФз

### 1. `mkPreparedSource` should support `go mod tidy` natively

The workspace тЖТ local deps transition creates inconsistent go.mod files. `mkPreparedSource` could:
- Run `go mod tidy` after patching go.mod with replace directives
- But needs network for transitive deps тЖТ won't work in sandbox
- Alternative: add a `postPatchExtra` that patches vendor/modules.txt instead

### 2. All LarsArtmann Go repos need `proxyVendor = true` audit

Every repo using `mkPreparedSource` with `go mod tidy` in `preBuild` is at risk. The pattern needs:
```nix
inherit proxyVendor;  # Critical!
overrideModAttrs = _: { preBuild = "go mod tidy"; };
preBuild = "go mod tidy";
```

### 3. Centralize vendor hash management

Currently each overlay manually specifies vendorHash. A hash registry or auto-hash mechanism would prevent cascade breakage on nixpkgs updates.

### 4. `.gitignore` audit for repos committing vendor/

The `credentials*` pattern in crush-daily is a cautionary tale. Any repo committing vendor/ needs `.gitignore` patterns scoped to root only.

### 5. Nixpkgs version pin in AGENTS.md

The behavior changes in nixpkgs 26.11 `buildGoModule` are significant. AGENTS.md should document which nixpkgs version introduced what changes.

## g) Top #1 Question I Cannot Figure Out Myself

**How should `mkPreparedSource` handle the "workspace transitive deps in vendor/modules.txt" problem architecturally?**

The core tension:
- `mkPreparedSource` patches go.mod with local `replace` directives
- `go mod vendor` then discovers transitive deps from those local sources
- Those transitive deps appear in `vendor/modules.txt` but NOT in `go.mod`
- nixpkgs 26.11 validates consistency тЖТ build fails
- `go mod tidy` fixes it but needs network (sandbox blocks)
- `proxyVendor = true` + `go mod tidy` in main build works (GOPROXY=file://) but feels hacky

The question: **Should `mkPreparedSource` itself run `go mod tidy` (requiring network/fixed-output), or should it generate a patch for go.mod that adds the missing indirect requires, or should the pattern always be `proxyVendor = true` + tidy in the build?** This is an architectural decision for `go-nix-helpers` that affects every LarsArtmann Go repo with workspace modules.
