05e) What We Should Improve

1. Backup system is a lie

The backup pipeline blindly pipes pg_dump to a file with no size validation, no integrity checks, and no off-site replication. It silently produces 0-byte files when databases are down and reports success. This is worse than no backup because it creates false confidence. Every backup script needs: size validation, error code checking, off-site replication, and periodic restore testing.

2. Single-device BTRFS is the worst of both worlds

BTRFS on a single device gives you checksum verification (so you know data is corrupt) but no recovery (because there's no mirror to repair from). You get the overhead of CoW checksumming with none of the resilience benefit. Either add a second device for BTRFS RAID1, or switch /data to ext4 (simpler, no false sense of security) and rely on proper backups instead.

3. Deploy gap is dangerous

The running generation is 9+ days behind git. OOM hardening, mount validation, Docker fixes — all committed but not running. The system that boots on next power cycle is not the system in git. Deploy needs to be unblocked and run immediately. The disk crisis that blocked it may be a chicken-and-egg: can't deploy because disk is full, disk is full because old generations aren't being cleaned.

4. Container health checks are lies

pg_isready only checks if the port is open, not if the database actually responds to queries. Docker reports containers as "healthy" when the DB is functionally broken. Health checks need to run actual queries (SELECT 1) or application-level checks.

5. Large modules need splitting

monitor365.nix (716 lines), signoz.nix (705 lines), forgejo.nix (583 lines) — these are too large to maintain safely. Each contains multiple concerns crammed together. Split by responsibility (database, container, backup, monitoring).

6. 178 status reports need archiving

docs/status/ has 178 files. The useful signal-to-noise ratio is dropping. Pre-session-100 reports should be archived to docs/status/archive/. An archive/ subdirectory already exists but isn't being used aggressively enough.

7. Dependency hygiene — stale go.sum/vendorHash workarounds

library-policy and mr-sync still need mkTidyOverride workarounds. 4 nixpkgs packages have doCheck = false overrides due to test failures. These are technical debt that should be fixed upstream.

8. No automated disk cleanup

nix.gc runs but Docker volume pruning is broken. 624 orphaned Docker volumes (37 GB) accumulated. Need a systemd timer that runs docker volume prune -f + nix-collect-garbage --delete-older-than 7d weekly.

07g) The #1 Question I Cannot Answer

What caused the BTRFS corruption — and is the NVMe drive itself failing?

The /data filesystem has 23.5 million uncorrectable checksum errors and is actively growing. But the NVMe SMART data is clean — no media errors, no reallocated sectors, no UNC errors at the hardware level. This is deeply unusual.

If the drive hardware is healthy, the corruption must be coming from one of: (a) a BTRFS bug triggered by the June 19 OOM crash / hard reset (BTRFS is known for CoW tree corruption on unclean shutdowns), (b) memory corruption that was written to disk (ECC would help but the Ryzen AI Max+ 395 doesn't have ECC), or (c) a failing controller or cable (SMART wouldn't necessarily catch this).

I cannot determine which of these is the root cause. Without knowing this, reformatting /data might just result in the corruption returning. The decision tree branches critically:

  • If (a) OOM-induced CoW corruption → reformat is safe, deploy OOM hardening first, then reformat
  • If (b) memory corruption → the drive is fine but RAM needs replacing; reformat + memtest86
  • If (c) hardware failure → the drive/controller/cable needs replacing before any data goes back

What I need from you: Have you run memtest86 since the corruption started? Do you have smartctl -a /dev/nvme0n1 output from before the corruption? Was the system overclocked or undervolted at any point? Is there a second NVMe slot to test with? This determines whether reformatting is a fix or a waste of time.

Why this matters more than anything else: You can fix backup systems, deploy OOM hardening, and reformat the filesystem — but if the underlying hardware or memory is silently corrupting data, none of it matters. Every new byte written to this system may be corrupted. This is the one thing I cannot diagnose or fix from the Nix configuration alone.
