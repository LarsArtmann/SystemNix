94G free of 723G. The last report showed 93%. It's improved (BTRFS snapshots may have expired), but nix-gc hasn't run on the deployed config (the btrfs-health guard is undeployed). Another big build could push it back up. The /nix/var/nix/builds sandbox cleanup timer is also undeployed.

3. SSO changes are completely untested on hardware

The Gatus and Forgejo OIDC configs eval-clean and render correct JSON, but OIDC flows have many runtime failure modes (redirect URI mismatch, cookie domain issues, PKCE errors, DynamicUser credential path permissions). Until a browser actually completes the passkey → callback → authenticated-session flow, this is unverified.

eWhat We Should Improve

1. Deploy cadence — stop accumulating 2-week gaps

73 undeployed commits is unacceptable. Either deploy more frequently, or maintain a "deploy readiness" dashboard showing the commit gap. The pre-deploy-check + deploy.sh infrastructure exists — use it.

2. Automated double-auth detection

The native-OIDC + protectedVHost double-auth conflict is a sharp footgun (caused redirect loops). A pre-deploy-check rule that flags any vHost using protectedVHost whose backend also has native OIDC would prevent recurrence.

3. SSO test automation

OIDC flows are brittle. A headless browser test (Playwright) that hits each Layer 1 vHost, completes the Pocket ID flow, and asserts an authenticated session would catch regressions at deploy time.

4. The immich/oauth2-proxy sops secrets are unguarded

In sops.nix, the immich and oauth2-proxy client secrets have service-specific owners but are NOT wrapped in lib.optionalAttrs (svcEnabled ...). On rpi3 (which enables sops-config but not immich), these would block ALL secrets atomically if the users don't exist. The gatus secret I added correctly uses the guard — the older ones are latent debt.

5. Backup strategy is undocumented and likely incomplete

Flagged in the c2951791 audit. No systematic backup of Pocket ID data, Forgejo repos, or sops-encrypted secret files off-host. DiscordSync has GCS backup; the critical identity/config layer does not.
