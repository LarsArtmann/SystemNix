The system is functional — but running on a dying filesystem.

05E) What We Should Improve

💥 BTRFS Single-Device = No Recovery

The entire /data partition is BTRFS on a single NVMe with no redundancy. BTRFS checksumming detected the corruption but couldn't fix it. If this were BTRFS RAID1 (or even BTRFS with periodic snapshots), recovery would be trivial. Single-device BTRFS gives you detection without recovery — the worst of both worlds.

💥 No Off-site Backup

Zero off-site backup exists. All backups are pg_dump files on the same machine. If the NVMe fails completely, everything is gone. The Hetzner StorageBox + BorgBackup research is done but never deployed. This is the #1 infrastructure gap.

⚠️ Backup Validation Missing

The backup system blindly pipes pg_dump to a file. If the DB is down, it creates a 0-byte file and moves on. No size check, no pg_restore --dry-run validation, no alerting. The backup timer should fail loudly when it produces empty output.

⚠️ Docker Volume Hoarding

628 volumes, 624 orphaned, 37 GB wasted. Docker Compose creates anonymous volumes that are never cleaned up. The docker-prune.timer runs weekly but apparently doesn't remove volumes. Need docker volume prune -f in the timer, or better: name all volumes explicitly and track lifecycle.

⚠️ Container Health Checks Are Lies

Manifest PostgreSQL reported "healthy" while every query was failing with I/O errors. The healthcheck (pg_isready) only checks if the port is open — it doesn't actually query the database. Twenty Redis reported "healthy" while its binary couldn't load SSL symbols. Healthchecks need to test actual functionality, not just network reachability.

💡 Pre-Deploy /data Snapshots

AGENTS.md says pre-deploy snapshots for /data are "manual only." This means any deploy that touches /data services has no rollback safety net for the data partition. Should be automated via btrbk or a pre-deploy hook.

💡 BTRFS Scrub Frequency Too Low

The scrub timer runs monthly (next: July 1). Given the corruption rate, this should be weekly. A weekly scrub would have caught this corruption days earlier, before it destroyed the databases.

❓G) The One Question I Cannot Answer

Should we reformat /data NOW, or wait until off-site backups are in place?

The /data filesystem is actively corrupting — 94,210 new errors in ~2 hours. Every hour we wait, more data is silently destroyed. But reformatting means:

  1. Downtime for every service on /data (Docker, models, AI, containers) — potentially hours
  2. Data migration risk — we'd be copying potentially-corrupted data to a backup, then restoring it. If the backup itself has corrupted blocks, we're back to square one
  3. No safety net — without off-site backup, a reformat mistake is catastrophic and irreversible

The counter-argument: waiting is also catastrophic. The corruption is eating data right now. The models, AI training data, and Docker volumes we haven't checked yet may be degrading as we speak. By the time we set up BorgBackup (which itself requires reading data from this corrupted filesystem), more data will be lost.

I cannot determine the right tradeoff here. Reformatting now stops the bleeding but risks losing everything if something goes wrong. Waiting preserves optionality but guarantees more data loss. The answer depends on: (a) how much of the 627 GB on /data is irreplaceable, (b) whether we can verify data integrity before reformatting, and (c) whether you have external copies of the critical data (models, etc.) that we can re-download.
